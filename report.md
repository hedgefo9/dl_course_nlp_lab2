## Задание на 4

### Бизнес-контекст
Собран proof-of-concept LLM-сервис для задачи распознавания SMS-спама в инфраструктуре сотового провайдера.

### Реализованные компоненты
- Docker-конфигурация: `docker-compose.yml`
- Docker образ на базе `ubuntu:22.04` + Python (без slim): `llm_service/Dockerfile`
- Запуск Ollama + загрузка модели + старт FastAPI в контейнере: `llm_service/entrypoint.sh`
- FastAPI-обертка над Ollama с одним endpoint: `llm_service/app/main.py` (`POST /generate`)
- Внешний скрипт для обращения к Ollama внутри контейнера: `scripts/call_container_ollama.py`
- Внешний скрипт для обращения к FastAPI-обертке: `scripts/call_fastapi_service.py`

### Как запускать
```bash
docker compose up -d --build
```

Проброс портов:
- `8001 -> 8000` (FastAPI)
- `11435 -> 11434` (Ollama API)

### Проверка из терминала (внутри контейнера)
Проверка списка моделей:
```bash
docker compose exec llm_service curl -s http://127.0.0.1:11434/api/tags
```
Результат: в списке присутствует `qwen2.5:0.5b`.

Проверка генерации через Ollama:
```bash
docker compose exec llm_service curl -s http://127.0.0.1:11434/api/generate \
  -d '{"model":"qwen2.5:0.5b","prompt":"Ответь одним словом: готов?","stream":false}'
```
Результат: модель вернула ответ (`"response":"Четвёртый."`), значит HTTP-инференс внутри контейнера работает.

### Проверка FastAPI-сервиса (с хоста)
```bash
curl -s http://127.0.0.1:8001/generate \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen2.5:0.5b","prompt":"Определи: это спам? SMS: WINNER!! Call now to claim your prize.","stream":false}'
```
Результат: FastAPI успешно вернул JSON-ответ от модели.

### Скрипт вне контейнера для обращения к Ollama внутри контейнера
```bash
python3 scripts/call_container_ollama.py \
  --prompt "Classify SMS as spam or ham: Congratulations! You have won a free vacation. Reply NOW."
```
Результат: получен корректный JSON от `http://127.0.0.1:11435/api/generate`, в ответе модель вернула `"spam"`.

## Дополнительная исследовательская часть (задание на 5)

### Что реализовано
- Добавлен скрипт исследования: `scripts/evaluate_prompting_techniques.py`.
- Реализованы 4 техники: `zero-shot`, `CoT`, `few-shot`, `CoT + few-shot`.
- Для техник `CoT`, `few-shot`, `CoT + few-shot` заданы отдельные системные промпты с требованием JSON-ответа:
  - `{"reasoning":"<строка>","verdict":<0 или 1>}`.
- В FastAPI-обертке (`llm_service/app/main.py`) добавлено поле `system`, чтобы системный промпт можно было передавать извне.
- Скрипт автоматически считает метрики `accuracy`, `precision`, `recall`, `f1`.

### Системные промпты (использованы в скрипте)

`CoT`:
```text
Ты классификатор SMS-спама.
Применяй технику Chain-of-Thought (CoT): перед финальным ответом последовательно оцени признаки:
1) срочный призыв к действию;
2) обещание выигрыша или выгоды;
3) запрос денег, персональных данных, перехода по ссылке или ответа на короткий номер;
4) признаки обычной личной переписки.

Верни только JSON и ничего больше:
{"reasoning":"<краткая строка>","verdict":<0 или 1>}
```

`few-shot`:
```text
Ты классификатор SMS-спама.
Используй few-shot подход и опирайся на примеры (spam/ham), затем верни:
{"reasoning":"<краткая строка>","verdict":<0 или 1>}
```

`CoT + few-shot`:
```text
Ты классификатор SMS-спама.
Используй CoT + few-shot:
1) внутренне оцени признаки спама;
2) сверяйся с примерами;
3) верни только JSON:
{"reasoning":"<краткая строка>","verdict":<0 или 1>}
```

Полные тексты промптов зафиксированы в `scripts/evaluate_prompting_techniques.py` и в `artifacts/prompting_metrics.json`.

### Как запускался эксперимент

```bash
python3 -u scripts/evaluate_prompting_techniques.py --samples-per-class 400 --max-attempts 2
```

Параметры прогона:
- Датасет: `spam.csv`
- Модель: `qwen2.5:0.5b`
- Endpoint: `http://127.0.0.1:8001/generate` (FastAPI -> Ollama)
- Сбалансированная выборка: 800 сообщений (`ham=400`, `spam=400`)

### Метрики

| Техника        | Accuracy | Precision | Recall |     F1 | Parse error rate |
|----------------|---------:|----------:|-------:|-------:|-----------------:|
| zero-shot      |   0.4938 |    0.4958 | 0.7350 | 0.5921 |           0.0000 |
| CoT            |   0.5487 |    0.5358 | 0.7300 | 0.6180 |           0.0262 |
| few-shot       |   0.6338 |    0.6474 | 0.5875 | 0.6160 |           0.0125 |
| CoT + few-shot |   0.5925 |    0.6045 | 0.5350 | 0.5676 |           0.0925 |

### Вывод

На этом прогоне `few-shot` дал лучшую `accuracy` (0.6338) и лучшую `precision` (0.6474).  
Лучший `f1` и `recall` у `CoT` (`f1=0.6180`, `recall=0.7300`), но у него ниже точность и выше доля ошибок формата, чем у `few-shot`.  
`CoT + few-shot` в текущей реализации оказался нестабилен по формату (`parse error rate=0.0925`) и по метрикам уступил `few-shot`.

Практический вывод: для этой модели наиболее прагматичный вариант — `few-shot`, если важнее общий баланс и аккуратность ответов.  
Если приоритет — не пропускать спам, можно рассматривать `CoT`, но тогда нужно дополнительно снижать ложные срабатывания и жёстко валидировать формат JSON.
